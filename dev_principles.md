## 2. 핵심 개발 원칙

### 원칙 1: 🎯 "시연 가능성 우선" (Demo-First)
- 완성도보다 **동작하는 시나리오 한 바퀴**를 최우선으로 한다.
- 앱 실행 → 음성 안내 → 포즈 인식 → 카운팅 → 완료 알림의 **E2E 플로우가 끊기지 않는 것**이 가장 중요하다.
- 각 기능의 완성도를 80%까지만 올리고, 나머지 20%는 시연 후 피드백으로 결정한다.

### 원칙 2: 🚫 "안 만드는 용기" (Ruthless Scoping)
- PRD의 **제외 범위(Out of Scope)**를 엄격히 지킨다.
  - ❌ 회원가입/로그인 → 고정 ID
  - ❌ 결제 시스템, 마이페이지
  - ❌ 관제 대시보드
  - ❌ 의료기기 수준 알고리즘
- 새로운 기능 요청이 들어오면 **"이것 없으면 시연이 불가능한가?"**로 판단한다.

### 원칙 3: 👴 "시니어 퍼스트 UX" (Invisible UX)
- 모든 인터랙션은 **음성(VUI) + 자동 인식**으로 설계한다.
- 화면 터치가 필요한 요소는 최소화하고, 불가피한 경우 **큰 버튼 + 고대비 색상**으로 구현한다.
- 카메라 프리뷰는 **블러 처리** → 프라이버시 보호 + 시각 소거 UX 적용.
- 3m 거리에서 읽을 수 있는 폰트 크기와 색상 대비를 기본으로 한다.

### 원칙 4: 🔌 "조립형 아키텍처" (Plug & Play)
- 직접 개발 < **검증된 외부 서비스 조합**을 우선한다.
  - AI 포즈 인식: Google Teachable Machine / MediaPipe Pose
  - TTS 음성: Typecast / ElevenLabs (사전 녹음 mp3)
  - 백엔드 자동화: Make (Integromat) + 웹훅
  - 알림: 비즈엠 카카오톡 알림톡 API
- 각 모듈은 **독립적으로 교체 가능**하게 느슨하게 연결한다.

### 원칙 5: ⏱️ "주 단위 검증" (Weekly Validation)
- 매주 마일스톤에 맞춰 **동작 가능한 산출물**을 만든다.
- 1주차: AI 모델 + 음성 에셋 → "인식이 되는가?"
- 2주차: UI + 핵심 로직 → "E2E 흐름이 돌아가는가?"
- 3주차: 외부 연동 → "카톡 알림까지 가는가?"
- 4주차: 현장 테스트 → "시니어가 혼자 할 수 있는가?"

### 원칙 6: 🧪 "실환경 테스트 기준" (Real-World Testing)
- 테스트는 항상 **3m 거리 + 거실 환경**에서 수행한다.
- 모니터/데스크 앞에서만 테스트하고 "잘 된다"고 판단하지 않는다.
- 밝은 조명 / 어두운 조명 / 역광 등 다양한 조건을 고려한다.

## 3. 코드 및 파일 컨벤션

| 항목 | 제안 |
|------|------|

| **커밋 메시지** | `feat:`, `fix:`, `docs:` 등 Conventional Commits |
| **브랜치 전략** | `main` 단일 브랜치 (1인 개발, MVP) |

## 4. 의사결정 기준 (Decision Filter)

새로운 기능/변경 요청 시 아래 체크리스트로 판단:

```
□ 이것 없으면 8/3 시연이 불가능한가?
□ 시니어가 터치 없이 사용할 수 있는가?
□ 이번 주 마일스톤 안에 들어가는가?
□ 외부 서비스로 대체할 수 있는가?
```

→ 하나라도 "아니오"면 **다음 버전으로 미룬다.**

## 5. 열린 질문 (함께 정해야 할 것들)

1. **기술 스택 확정**: FlutterFlow vs Bubble vs 순수 웹(HTML+JS+MediaPipe)?
2. **Git 사용 여부**: 버전 관리를 Git으로 할지, 노코드 플랫폼의 내장 버전 관리에 맡길지?
3. **에러 핸들링 수준**: MVP에서 에러 발생 시 어디까지 처리할지?
4. **음성 파일 관리**: mp3를 앱 내 번들로 넣을지, 클라우드(S3 등)에서 스트리밍할지?
5. **테스트 기록 방식**: 주차별 테스트 결과를 어떤 형태로 남길지?
